# SAN 精力系统

> **一句话理解**：机器人也会累，精力不够时会变得"懒得说话"。

---

## 什么是 SAN 系统？

SAN（Sanity）系统模拟了机器人的"精力值"：

- 每次处理消息会**消耗**精力
- 每小时自动**恢复**精力
- 精力低时机器人会变得"疲惫"

这让机器人的行为更加生动有趣。

---

## 精力如何变化

### 消耗

| 行为 | 消耗 |
|------|------|
| 处理一条消息 | `san_cost_per_message`（默认2点） |

### 恢复

| 方式 | 恢复量 |
|------|--------|
| 每小时自动恢复 | `san_recovery_per_hour`（默认10点） |

### 群氛围影响

| 场景 | 影响 |
|------|------|
| 高活跃群 | +5 额外恢复 |
| 低活跃群 | -3 额外消耗 |
| 正向氛围 | +3 额外恢复 |
| 负向氛围 | -5 额外消耗 |

---

## 精力状态

```
SAN 100 ──────────────────────── 精力充沛
       │
SAN 50  ────── 正常运作
       │
SAN 20  ────── ⚠️ 低精力阈值
       │
SAN 0   ────── 精疲力尽，可能拒绝服务
```

### 不同状态的表现

| 状态 | SAN值 | 表现 |
|------|-------|------|
| 充沛 | 80-100 | 正常回复，积极主动 |
| 正常 | 20-79 | 正常回复 |
| 疲惫 | <20 | 回复简短，可能说"累了" |
| 耗尽 | 极低 | 可能拒绝服务 |

## 使用示例

### 示例 1：查看精力值

```
用户：/san show
机器人：当前SAN值：45/100，状态：正常，恢复速率：10/小时
```

### 示例 2：精力不足时的表现

```
[机器人SAN值低于20]

用户：帮我写个代码
机器人：我现在有点累了，脑容量超载了...让我休息一会再帮你好吗？
```

### 示例 3：手动恢复（管理员）

```
/san set 100
```

## 常见问题

### Q: 机器人总是说"累了"怎么办？

**原因**：SAN 值消耗过快或上限太低。

**解决方案**：

**方案1：提高上限**
```json
{
  "san_max": 150
}
```

**方案2：降低消耗**
```json
{
  "san_cost_per_message": 1.0
}
```

**方案3：提高恢复速度**
```json
{
  "san_recovery_per_hour": 20
}
```

**方案4：关闭 SAN 系统**
```json
{
  "san_enabled": false
}
```

---

### Q: 精力值什么时候恢复？

每小时自动恢复一次，不需要任何操作。

例如：
```
当前SAN：50
1小时后：60
2小时后：70（达到上限100后停止）
```

## 相关配置

```json
{
  "san": {
    "san_enabled": true,              // 启用SAN系统
    "san_max": 100,                   // 精力上限
    "san_cost_per_message": 2.0,      // 每条消息消耗
    "san_recovery_per_hour": 10,      // 每小时恢复
    "san_low_threshold": 20,          // 低精力阈值
    "san_auto_analyze_enabled": true, // 启用群氛围分析
    "san_analyze_interval": 30        // 分析频率（秒）
  }
}
```

## 设计目的

SAN系统的存在不是为了限制使用，而是为了：

1. **增加真实感**：机器人也会"累"，更像一个真实的伙伴
2. **调节活跃度**：防止机器人在过于活跃的群里变成"话痨"
3. **增加策略性**：用户需要考虑机器人的"状态"来决定何时互动

如果你觉得这个系统影响了使用体验，随时可以在配置中关闭它。
